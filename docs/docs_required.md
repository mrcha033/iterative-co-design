**한 줄 요약**
코드 구조(엔지니어링)와 실험(연구) 각각에 대해 “무엇을, 왜, 누가, 완료기준은 무엇인가”가 분명한 문서 세트를 제안한다. 불필요한 말 제거, 실행 가능성과 재현성 확보 중심.

---

## 가정·제약

* 범위: HW–SW 공동설계(모듈러리티 기반 재퍼뮤테이션, 스펙트럴+로컬 서치, sparsity/quant/KV 캐시) 코드화 및 실험.
* 산출물 용도: 리포지토리 루트에 `/docs/`로 포함, 각 문서는 **소유자/리뷰어/완료기준**(Acceptance Criteria) 명시.
* 톤: 비판적·실무 중심(과장 금지), 결정은 **ADR(Architecture Decision Record)** 로 남김.

---

## 산출물

### Ⅰ. 코드 구조 구축을 위한 문서(엔지니어/PM 시각)

1. **제품 요구사항 문서(PRD)**

* 내용: 문제정의(지표·제약), 사용자/시나리오, 범위/비범위, 성공메트릭(latency/EpT/L2hit 개선 %), 릴리즈 마일스톤.
* 완료기준: 모든 하위 설계 문서와 지표가 PRD 메트릭과 추적가능(Traceability).

2. **시스템 아키텍처 명세(SAS)**

* 내용: 상위 블록도(모델·컴파일러·런타임·커널·하드웨어), 데이터플로/컨트롤플로, 성능·메모리·전력 예산, 오류/장애 모델.
* 완료기준: 컴포넌트 간 경계(계약)와 리소스 예산이 수치로 고정.

3. **인터페이스 제어 문서(ICD) / API 스펙**

* 내용: Python/CLI/라이브러리 API, 입출력 타입, 예외/에러코드, 역호환 정책(semver).
* 완료기준: 파괴적 변경 시 버전 정책·마이그레이션 포함.

4. **IR/다이얼렉트 스펙**

* 내용: IR(StableHLO/TVM 등)에서 쓰는 **속성·메타데이터(레이아웃, 블록, permutation, sparsity tag)** 정의, Lowering 규칙, Shape/Dtype 추론 규칙.
* 완료기준: 예제 IR 스니펫과 Pass 파이프라인에서의 **삽입 지점** 명시.

5. **컴파일러 패스 설계서(Pass Design Doc)**

* 내용: Pass 입력/출력 계약, 비용모델(모듈러리티·cache-cost), 분석/변환 순서, 실패/보정 경로, Pass 간 의존성 DAG.
* 완료기준: 각 Pass에 단위테스트 전제조건과 황금샘플(golden) 고정.

6. **비용모델/코스트 함수 명세**

* 내용: `C(π)`(cache proxy), 모듈러리티 Q, 측정치 맵핑(L2 hit ↔ 추정식), 파라미터/가중치 튜닝 규칙.
* 완료기준: 유닛테스트로 **단조/경향** 검증(예: 블록 구조 강할수록 C↓, Q↑).

7. **그래프 생성 규격(Graph Construction Spec)**

* 내용: 공접근 가중치 W 생성(실모델·mock 각각), 노이즈·정규화, 희소 표현(압축 포맷).
* 완료기준: 동일 입력에서 결정론적 W 재생산.

8. **Permutation 솔버 설계**

* 내용: 스펙트럴 초기화, 로컬탐색(2-opt/adj-swap)·정지조건, 스케일전략(부분 고유값, 샘플링), 병렬화.
* 완료기준: 시간복잡도/메모리 상한, 대규모 입력에서 열화 패턴 문서화.

9. **상태-변환 모듈 스펙(S/Q/K)**

* 내용: sparsity(2:4/HDS, unstructured), PTQ/FP8, KV 캐시 압축의 **연산/메모리 효과** 및 역전파되는 메타데이터.
* 완료기준: 각 변환 후 **필수 재퍼뮤테이션 트리거 조건** 명문화.

10. **커널 계약서(Kernel Contract)**

* 내용: 커널별 텐서 레이아웃/얼라인먼트, vectorization/warp 매핑, prefetch 정책, 캐시/공유메모리 사용.
* 완료기준: Nsight/rocprof 지표와 계약의 대응표.

11. **런타임/메모리 관리 설계**

* 내용: KV 캐시 관리자, 핀드 메모리, 그래프 캡처, 스트림/이벤트, OOM/프래그멘테이션 처리.
* 완료기준: 스트레스 테스트 시나리오 통과 기준.

12. **관측성/로깅 스펙(Observability)**

* 내용: 성능 카운터, 프로파일링 훅, 이벤트 스키마, 샘플링 비율, 개인정보/보안 가드.
* 완료기준: 오버헤드 상한(예: <1%) 충족.

13. **보안·라이선스·공급망(SBOM)**

* 내용: 서드파티 라이선스, 취약점 정책, 재현 빌드(locks), SLSA 레벨 목표.
* 완료기준: CI에서 SBOM 생성·스캔 자동화.

14. **테스트 전략서(Test Plan)**

* 내용: 단위/통합/E2E/성능/회귀/결정론/숫자안정성, mock vs 실제, seed 정책.
* 완료기준: 필수 커버리지·성능 회귀 게이트 정의.

15. **성능 벤치/튜닝 가이드**

* 내용: 실험 벤치 구성, 파라미터 튜닝 체크리스트, 병목 식별 절차.
* 완료기준: 최소 재현 표준(토큰/s, L2 hit, EpT) 자동 리포트.

16. **릴리즈/변경관리 계획(Release & ADR)**

* 내용: 버전, 마이그레이션, 호환성, 중요 결정의 ADR 로그.
* 완료기준: 각 릴리즈에 대응되는 ADR 링크 완비.

17. **기여/코딩 규약(Contrib/Style)**

* 내용: 코딩스타일, 포매터/린터, 브랜치/리뷰, 대형 변경 절차.
* 완료기준: 첫 외부 PR이 문서만으로 통과 가능.

### Ⅱ. 실험을 위한 문서(연구자 시각)

A) **실험 설계서(Study Protocol)**

* 내용: 가설·메커니즘(모듈러리티→L2→Latency/EpT), 독립/종속변수, 통제요인, 실험 매트릭스, 중단·피벗 기준.
* 완료기준: 누구나 문서만 보고 동일 실험표 구축.

B) **평가지표 정의서(Metrics Spec)**

* 내용: Latency·tokens/s·L2 hit·EpT 정의/단위/측정법, 신뢰구간·효과크기, 유의수준.
* 완료기준: 지표별 측정 오차·검정 방법 명시.

C) **측정 SOP(Standard Operating Procedure)**

* 내용: Nsight/NVML/전력계 사용법, 샘플링 주기, 워밍업/고정클럭, 반복 횟수, 노이즈 처리.
* 완료기준: 서로 다른 운영자가 ±X% 내 재현.

D) **데이터셋/모델 카드**

* 내용: 버전·라이선스·전처리·분할·한계, 모델 체크포인트/토큰화 버전.
* 완료기준: Commit hash·다운로드 경로 고정.

E) **실험 환경 명세(Environment Manifest)**

* 내용: 하드웨어(A100/H100 등), 드라이버, 프레임워크 버전, 컨테이너/Dockerfile, RNG seed 정책.
* 완료기준: 단일 스크립트로 환경 재구성.

F) **아블레이션 계획(Ablation Plan)**

* 내용: Dense, S-only, P-only, Linear, **Iterative(S↔P)**, 변환 강도(희소율/정밀도) 스윕.
* 완료기준: 각 축별 최소 비교 세트와 기대 방향성 명시.

G) **통계·검정 계획(Stat Plan)**

* 내용: 샘플 수, CI/부트스트랩, 정규성 가정 여부, 다중비교 교정.
* 완료기준: 표/그림에 p/CI·효과크기 병기.

H) **재현 패키지(Repro Pack)**

* 내용: `make repro` 전 과정(다운로드→전처리→실험→리포트), 로그/아티팩트 경로, 시간·비용 추정.
* 완료기준: 외부 검증자가 보고서 재생산 성공.

I) **결과 리포트 템플릿**

* 내용: 표(성능/에너지), 그림(Pareto, Q–L2–Latency 상관), 실패 사례·한계.
* 완료기준: 자동 생성 스크립트 포함.

J) **위협요인/한계(Threats to Validity)**

* 내용: 하드웨어 종속성, 측정 바이어스, 데이터 드리프트, 초기 순열 의존성.
* 완료기준: 완화책과 잔여 리스크 명시.

K) **윤리/컴플라이언스 메모**

* 내용: 라이선스·개인정보·보안, 책임 있는 보고(에너지·탄소 각주 가능).
* 완료기준: 공개 전 체크리스트 통과.

L) **Artifact Evaluation(학회 AE) 체크리스트**

* 내용: 계정/자원 요구, 실행 시간, 평가 스크립트, 기대 출력 샘플.
* 완료기준: AE 폼 항목과 1:1 매핑.

M) **클레임–증거 추적표(Claim–Evidence Matrix)**

* 내용: 논문 주장별 증거(표/그림/로그) 링크, 반례·감쇠 조건.
* 완료기준: 모든 주요 주장에 근거 연결.

N) **그림/표 스타일 가이드**

* 내용: 단위·축 레이블·유의성 표기 규칙, 색상/선택 기준.
* 완료기준: 전 결과물 시각 일관성.

## 테스트(검증 방식)

* **문서 린트**: 각 문서에 `Owner/Reviewer/Acceptance` 필드 필수. CI로 누락 검사.
* **드라이런**: 신규 팀원이 문서만 보고 “mock 실험” 재현 → 통과 못하면 문서 보강.
* **추적성 테스트**: PRD 메트릭 ↔ 코드/실험 결과 링크 자동 점검.

---

## 성능 메모/리스크/대안

* **리스크**: 문서 과잉은 속도 저해.
  **대안**: 초기엔 굵은 문서 7종(PRD, SAS, ICD, Pass Doc, Cost Spec, SOP, Repro Pack)만 생성 → 나머지는 스프린트별 증분.
* **리스크**: IR·Pass 설계 위계 불명확 → 구현 혼선.
  **대안**: Pass 파이프라인 DAG와 입력/출력 계약을 **ICD 수준**으로 명문화.
* **리스크**: 측정 일관성 실패.
